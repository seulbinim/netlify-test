# UI Project 회고록  

## 1. 프로젝트 개요  
- **프로젝트명**: [TAING UI 제작]  
- **기간**: 2025.03.14 ~ 2025.03.25  
- **팀원**: 엄현욱, 우영찬, 이진현, 김지연  
- **사용 기술**: HTML, Vite, TailwindCSS  
- **목표**:  
  - 기존 웹사이트와 비슷한 UI까지 제작  
  - 최대한 JavaScript를 사용하지 않고 CSS로만 제작하고자 함

---

## 2. 프로젝트 과정  
### 기획  
- TAING 웹사이트의 반응형 UI가 다양한 형태로 변화하는 과정을 직접 경험해 보고자 선정하였다.
- 기존 TAING 웹사이트와 차별화된 새로운 메인 컬러를 지정하여 디자인적 차별성을 부여했다.
- UI를 효율적으로 구성하기 위해 컴포넌트의 범위를 정의하고, 이를 조립하여 개발을 진행하기로 했다.
- 기존 Figma 시안에서 메인 컬러, 썸네일 등을 수정하여 새로운 Figma 시안으로 구성했다.
- 사용자의 경험(UX)과 접근성을 고려하여, 사용자가 최대한 쉽게 이용할 수 있도록 코드를 구성했다.

### 과정  
-  프로젝트는 유지보수성을 높이고자 모듈화된 컴포넌트 기반으로 설계하여 재사용성과 확장성을 고려했다.
-  아래와 같이 3개의 마일스톤을 생성하여 진행하였다.
   1. 개발 환경 세팅 
   2. 컴포넌트 생성 
   3. 페이지 생성
-  주요 컴포넌트는 헤더, 영상 주제별 컴포넌트, 푸터, 버튼, 로그인 입력 등으로 나누었다.
-  페이지를 조립할 때 접근성까지 고려하여 기존에 생성한 컴포넌트에 추가하면서 조립하였다.
-  일부 페이지를 생성할 때 필요한 JavaScript 코드는 AI의 도움을 받아 적용하였다.

### 배포
- 배포 환경: ***Netlify***

## 3. 어려웠던 부분과 해결 방법  
| No. | 문제점 | 원인 | 해결 방법 |
| ---- |-------|------|----------|
| 1 | absolute 속성이 적용 안됨 | `<div>`태그로 무작정 감싸기 | 처음 작성한 코드부터 하나씩 부모자식 상속의 관계를 하나하나 생각해보며 반영하여 원하는 스타일링을 적용함 |
| 2 | text색상 이슈 | 배경색에 따른 미묘한 차이를 가진 text 색상 | Figma 시안과 같은 text 색상이었지만 조금 다르게 보이는 현상이 발생하여 배경색상 또한 완전 동일한 색상으로 부여하여 수정함 |
| 3 | 반응형 이슈 | `1920px` 이상으로 늘렸을 때 포스터 2개가 동시에 나타남 | TAING오리지널 작업에서 반응형에 따라 xl이상일 때 두개의 이미지가 한번에 들어가 md와 xl사이인 lg:1280px을 부여하여 수정함 |
| 4 | 이미지 위에 배경색 설정 | 이미지에 배경색을 지정하는게 아닌 `<svg>`태그 내부에 있는 `<rect>`에 지정 | 프로필이 잠겨있다는 설정을 할 때 이미지 배경색을 어둡게 지정해주어야 하는데 Ai의 도움을 받아 이와 같이 수정함 `<rect width="114" height="114" rx="4" fill="rgba(0, 0, 0, 0.5)" />` |
| 5 | 검색 창 이슈 | 검색 창 컴포넌트에 input 요소가 가운데 배치 되지 않음 | `<div>`로 한 번 더 감싼 후 `class="flex items-center justify-center"` 속성을 부여함  |
| 6 | `<a>` 태그의 영역이 `<div>`태그 전체를 차지하는 경우 | 부모 `<div>`가 `width: 100%`를 가지면 `<a>` 태그가 이를 상속받아 전체를 차지함  | 부모 요소인 `<div>`에 `w-fit` 속성을 넣어서 해결함  |

## 4. 느낀점  
- 예전부터 '정말 GitHub에 나만의 저장소가 있었으면 좋겠다.'라는 생각만 가지고 있었다.
  그래서 Git을 처음 배울 때 생소하긴 했지만, 정말 열심히 배워보려고 했었던 것 같다.
  그 덕분에 지금은 조금 익숙해진 것 같다.

  협업 과정을 처음 설명만 들었을 때는 '와.. 이걸 어떻게 활용해서 협력을 해야 할까..?'라는 막막한 생각만 들었지만,
  실제 팀원들과 사용해 보니 GitHub의 협업 방식이 생각보다 체계적이고 효율적이라는 것을 깨닫게 되었다.

  각자 맡은 기능을 개별적으로 브랜치에서 작업한 후, pull request를 통해 코드 리뷰를 거쳐 병합하는 과정이
  단순히 코드 관리뿐만 아니라 팀원 간의 소통과 이해를 높이는 데도 큰 도움이 되었다.

  처음에는 익숙하지 않아 실수도 있고 충돌도 있었지만, 점점 Git의 다양한 기능과 협업 방식을 익히면서
  버전 관리의 중요성과 GitHub가 주는 편리함을 몸소 느낄 수 있었다.

  팀원들 모두 각자 맡은 바를 열심히 최선을 다해 준 결과, 마일스톤에 등록한 작업들을 기간보다 여유롭게 끝낼 수 있었다.
  밤낮 상관없이 서로 질문도 하며 공유하고 피드백했던 모든 순간이 의미 있었고, 보람찼다.
  10일 정도 되는 기간 동안 너무나 고생한 팀원들 모두가 자랑스럽다. **고생하셨습니다~~~!!**

## 5. 개선할 점 및 다음 목표  
### 개선할 점  
- 다양한 CSS 속성들이 아직까지는 완벽하게 적응되지 않는다.
  익숙해질 때까지는 다양한 방법으로 스타일을 짜보면 좋을 것 같다.
  그래도 UI 프로젝트를 진행하면서 다양한 속성을 사용하다 보니 전보다 더 성장한 것 같은 느낌도 받았다.

- UI 프로젝트이다 보니 CSS만 사용해서 페이지를 구현하는 것에 한계점이 있었던 것 같다.
  프로젝트가 끝나고 JavaScript 수업을 듣고 연습하여, 다음 프로젝트에는 JavaScript까지 적용해
  더욱 완성도 높은 결과물을 만들고 싶다.

- 다음 프로젝트에서도 Tailwind CSS를 사용해서 스타일링하고 싶다.
  Tailwind를 일주일도 배우지 않고 거의 바로 실전에 투입된 느낌이라 아직 CSS보다는 어색하긴 하지만,
  조금 더 익숙해진 상태에서 사용하면 어떻게 더 섬세하고 편리하게 작업할 수 있을지 기대된다.
